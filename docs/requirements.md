# VTuberSystemBase 要件定義書

本ドキュメントは、VTuber 向けリアルタイム 3D モーションキャプチャシステムの基盤である
本リポジトリ（VTuberSystemBase）の要件を定義するものである。

- 版数: 1.0（初版）
- 作成日: 2026-04-24
- 対象: VTuberSystemBase（UPM パッケージ + Unity プロジェクト一式）

---

## 1. 本システムの位置づけ

### 1.1 目的

VTuber 向けのリアルタイム 3D モーションキャプチャシステムを構築するための **共通基盤**
を提供する。本リポジトリを利用する各社・各スタジオは、本リポジトリが提供する UPM
パッケージを自社プロジェクトにインポートし、キャラクターのアバター・ステージアセット・
UI スキンなどを差し替えて運用することを想定する。

### 1.2 成果物の形態

- **UPM パッケージ**: 各社の Unity プロジェクトに組み込み可能なパッケージとして提供する。
- **リファレンス Unity プロジェクト**: パッケージの開発・検証を行うための Unity プロジェクトを本リポジトリ内に同梱する。

### 1.3 想定ユースケース

1. **スタンドアロンアプリ運用**: リファレンスプロジェクト（あるいは各社プロジェクト）を
   ビルドして配信現場で使用するケース。
2. **Unity Editor 上での運用**: Unity Editor の PlayMode で直接運用するケース。
   機能上の差異を最小化し、両モードで同じ体験を提供する。

### 1.4 本要件定義書のスコープ外

- アバターアセット、ステージアセット、UI スキン等の個別アセットそのものの仕様
  （これらは利用者側の責務）。
- MoCap ハードウェア（トラッカー、カメラ等）の選定や設定。
- 配信ソフト（OBS 等）への取り込み方法の詳細。

## 2. 対象環境・前提条件

### 2.1 開発環境

| 項目 | 値 |
| --- | --- |
| Unity バージョン | Unity 6.3 |
| レンダリングパイプライン | URP（Universal Render Pipeline） |
| UI フレームワーク | UI Toolkit（UIElements）を原則とする |

### 2.2 実行環境

| 項目 | 値 |
| --- | --- |
| 対応 OS | Windows（x86 アーキテクチャ） |
| ディスプレイ構成 | **2 画面以上** のマルチディスプレイ構成を必須とする |
| 配信連携 | 出力映像がそのまま YouTube 等の配信に載ることを前提とする |

### 2.3 運用形態

- 本システムの映像アウトプットは、キャプチャソフト（OBS 等）で取り込まれ、
  **編集を経ずそのまま配信に載る** ことを想定する。
- したがって、メイン出力画面に操作 UI・デバッグ情報・警告ダイアログ等が
  不用意に映り込んではならない。

## 3. システムアーキテクチャ要件

### 3.1 UI 画面とメイン出力画面の物理的分離

本システムは 2 つ以上のディスプレイを使い分ける。

| 役割 | 表示ディスプレイ | 内容 |
| --- | --- | --- |
| オペレーター UI | **1 画面目（Display 1）** | オペレーター向け操作 GUI |
| メイン出力 | **2 画面目以降（Display 2+）** | 配信に載るキャラクター + ステージ映像（全画面表示） |

- 「メイン出力」を配信ソースとして扱うため、メイン出力ディスプレイには
  UI 要素を **一切** 描画しないこと。
- 出力映像の切り替え（どの物理ディスプレイをメイン出力にするか等）には、
  並行開発中の [RuntimeDisplaySelector](https://github.com/Hidano-Dev/RuntimeDisplaySelector)
  パッケージを採用する。

### 3.2 UI プロセスとメイン出力の論理的分離

「UI 部分」と「メイン出力」は、同一プロセス内であっても **システム的に分離** し、
**LocalHost 経由のプロセス間通信** で制御する構造とする。

#### 3.2.1 背景（なぜ分離するか）

将来的に、以下の拡張を想定しているため、最初から通信経由で疎結合にしておく。

- LAN 内のタブレット端末からの遠隔操作
- ブラウザベースの WebUI（HTML/JS）からの操作
- 別 PC に UI を逃がしての運用

#### 3.2.2 通信方式

- トランスポート: LocalHost 上の TCP/UDP ソケットまたは WebSocket を想定
  （最終方式は設計フェーズで確定）。
- カメラデータの伝送は後述の通り **OSC** を採用する。
- すべての状態変更イベントは通信経由で発行し、UI から直接メイン出力側の
  オブジェクトを触らないこと。

### 3.3 ランタイム一貫性

- スタンドアロンビルドと Unity Editor PlayMode の両方で、
  上記のマルチディスプレイ・プロセス間通信アーキテクチャが機能すること。

## 4. UI シェル要件（Display 1 側）

### 4.1 タブ構成

Display 1 の UI は、以下の 3 つのタブを持つシングルウィンドウで構成する。

1. **キャラクター選択・設定画面**
2. **ステージ・ライティング・グローバル Volume 設定画面**
3. **カメラ・カメラ Volume 操作画面**

### 4.2 タブ切り替えの応答性要件（重要）

- **3 つのタブはアプリ実行中いつでも相互に行き来できること。**
- タブ切り替え時に **メイン出力が 1 フレームでもフリーズしてはならない**。
- これを担保するため、以下を遵守する。
  - **可能な限り起動時にすべての UI アセット・UI Document をロードしておく**。
  - どうしても新規ロードが必要な AssetBundle 等がある場合、
    **必ず別スレッドで非同期読み込み** を行い、メインスレッドをブロックしないこと。
  - 読み込み中に操作が必要な UI は、プレースホルダ表示や非活性化で待機させる
    （メイン出力のレンダリングには影響させない）。

### 4.3 UI 実装方針

- 原則として **UI Toolkit** でデザイン・実装する。
- UXML / USS / C# の分離原則に従う。
- 各タブは独立した UI Document（VisualTreeAsset）として実装し、
  起動時にすべて生成・アタッチし、表示/非表示のみで切り替える
  （都度インスタンス化しない）。

## 5. 機能要件

### 5.1 タブ 1: キャラクター選択・設定画面

#### 5.1.1 目的

MoCap アクター（演者）に、使用するアバター／プロファイルを割り当てるための画面。

#### 5.1.2 UI デザイン方針

- 一般的なゲームの **キャラクター選択 UI** を参考にデザインする。
- MoCap アクターを「ゲームのプレイヤー」に見立てる。
  - 複数アクター（Slot）と複数アバター候補を視覚的に紐付けられるレイアウトを採る。

#### 5.1.3 コア実装の依存

- キャラクターの選択・設定のコア実装は以下のパッケージを採用する。
  - [RealtimeAvatarController](https://github.com/Hidano-Dev/RealtimeAvatarController)
  - Slot ベースの MoCap ソース管理、アバタープロバイダ抽象化、
    モーションパイプラインを提供する。
- 本システムは RealtimeAvatarController の API を呼び出す **UI レイヤー** に
  責任範囲を限定し、MoCap 処理そのものは RealtimeAvatarController に委譲する。

#### 5.1.4 機能

- MoCap アクター（Slot）一覧の表示。
- 各 Slot に対するアバター候補の割り当て／変更。
- 各アバターごとの個別設定（RealtimeAvatarController が提供する設定項目を UI に
  露出する）。

### 5.2 タブ 2: ステージ・ライティング・グローバル Volume 設定画面

#### 5.2.1 目的

シーン全体のルック（見栄え）を確定するための画面。配信前の「画作り」フェーズで使用する。

#### 5.2.2 カメラ操作

- 本画面では **引きのカメラ 1 点** を操作しながらシーン全体の見え方を調整する。
- カメラ操作は [SceneViewStyleCameraController](https://github.com/Hidano-Dev/SceneViewStyleCameraController)
  を活用し、Unity Editor の Scene ビューと同等の直感的操作（マウス回転・パン・ズーム等）
  を提供する。

#### 5.2.3 ステージ

- ステージデータの読み込み・切り替えを本画面から行う。
- ステージは差し替え可能なアセット（Prefab / AssetBundle 等）として扱う。

#### 5.2.4 ライティング

- 任意の数の `Light` コンポーネントを **実行中に動的に生成・削除** できること。
- 各 Light について、以下を GUI で調整できること。
  - 角度（向き、Transform Rotation）
  - 色（Color）
  - 光の強さ（Intensity）
  - （必要に応じて）Type、Range、Spot Angle 等の追加パラメータ
- 生成した Light 構成は、後述の永続化要件に従い保存・復元できること（設計フェーズで詳細化）。

#### 5.2.5 グローバル Volume

- URP の Volume（Global Volume）上の各 Override（Bloom, Tonemapping 等）を
  GUI から編集できること。
- 編集結果はリアルタイムで Display 2+ のメイン出力に反映されること。

### 5.3 タブ 3: カメラ・カメラ Volume 操作画面

#### 5.3.1 目的

本番配信中の **カメラスイッチャー** UI。どのカメラの映像をメイン出力に流すかを切り替える。

#### 5.3.2 依存パッケージ

- [UniversalCamerawork](https://github.com/Hidano-Dev/UniversalCamerawork)（UCAPI）
  - カメラ状態（位置、回転、光学パラメータ、タイムコード等）の共通フォーマット
    （Flat Record / MessagePack）を提供する C++ DLL。
  - 本システムは UCAPI4Unity（UPM パッケージ）経由で利用する。
- [SceneViewStyleCameraController](https://github.com/Hidano-Dev/SceneViewStyleCameraController)
  - 本画面のカメラ操作 UX に利用する。

#### 5.3.3 スコープ（現フェーズ）

カメラスイッチャー機能は将来的により高性能なものへ差し替える予定があるため、
本フェーズでは以下の **4 つの処理ライン** までをスコープとする。

1. **入力**: SceneViewStyleCameraController で直感的にカメラを操作する。
2. **データ化**: 操作中のカメラ状態を、UniversalCamerawork で定められた
   規格（UCAPI Flat Record）でシリアライズする。
3. **伝送**: シリアライズしたカメラデータを **OSC** で LocalHost にリアルタイム送信する。
4. **適用**: LocalHost で受信したカメラデータを、各メイン出力ディスプレイの
   Camera コンポーネントに適用する。

#### 5.3.4 本フェーズでの非目標

- 複雑なトランジション（ディゾルブ、カット切替時の補間）
- マルチカメラのプレビュー／プログラム出力の同時管理
- 外部ハードウェアスイッチャー連携
- タイムライン録画・リプレイ

これらは次フェーズ以降の拡張として位置づけ、本フェーズの設計は
**「差し替え可能な最小機能」** を意識すること。

#### 5.3.5 カメラ Volume

- 各カメラに紐づく URP Volume（Local Volume）を UI から編集できること。
- カメラ切替時に、対応する Volume の効果が自動的に切り替わること。

## 6. 非機能要件

### 6.1 パフォーマンス

| 項目 | 要件 |
| --- | --- |
| メイン出力フレームレート | 配信解像度（1080p 想定）で安定して動作すること（具体値は設計フェーズで確定） |
| タブ切替時のフレーム落ち | **0 フレーム**（メイン出力側に影響しないこと） |
| AssetBundle 読み込み | メインスレッドをブロックしないこと（別スレッド必須） |

### 6.2 配信適合性

- メイン出力ディスプレイには **UI・デバッグログ・エラーダイアログを一切描画しない**。
- 重大エラー発生時も、メイン出力側は現状描画を維持し、UI 側（Display 1）にのみ通知する。

### 6.3 拡張性

- UI ↔ メイン出力の通信は LocalHost 経由で抽象化されているため、
  将来的に別端末の UI クライアント（タブレット、ブラウザ）へ置き換え可能であること。
- カメラスイッチャーは差し替え前提の構造であること（5.3.4 参照）。
- 利用者がアバター・ステージ・UI スキンを差し替えられること
  （UPM パッケージとして提供する本リポジトリの公開 API を明確化する）。

### 6.4 保守性

- 各機能モジュールは可能な限り独立したアセンブリ定義（asmdef）に分離する。
- 外部パッケージ依存（RealtimeAvatarController, SceneViewStyleCameraController,
  UniversalCamerawork, RuntimeDisplaySelector）はバージョン固定で管理する。

## 8. 開発フェーズ・スコープ分け

### 8.1 本フェーズ（Phase 1）で実装する範囲

- UI / メイン出力の論理分離アーキテクチャと LocalHost 通信基盤
- UI Toolkit による 3 タブ構成のシェル UI
- タブ 1: RealtimeAvatarController を組み込んだキャラクター選択・設定画面
- タブ 2: ステージ読み込み・Light 動的管理・Global Volume 編集
- タブ 3: SceneViewStyleCameraController + UCAPI + OSC + 適用までの一連ライン（5.3.3）

### 8.2 本フェーズで後回しにする範囲

- **RuntimeDisplaySelector 連携部分**（並行開発中のため）。
  本フェーズでは開発状況を随時フェッチして仕様を確認し、組み込みポイント（インタフェース）
  のみ先に切っておく。

### 8.3 次フェーズ以降の拡張候補

- 高機能カメラスイッチャーへの差し替え（トランジション、マルチカメラ PVW/PGM 等）
- LAN 内タブレット UI
- ブラウザベース WebUI
- タイムライン録画・リプレイ
- 配信ソフト連携の高度化

## 9. 用語定義

| 用語 | 意味 |
| --- | --- |
| メイン出力 | Display 2 以降に全画面表示される、配信に載るキャラクター + ステージ映像 |
| UI | Display 1 に表示されるオペレーター用 GUI |
| Slot | RealtimeAvatarController における MoCap ソース管理の単位。本システムでは「アクター枠」として扱う |
| MoCap アクター | MoCap 機材を装着する演者。本システムでは Slot と 1:1 対応 |
| UCAPI | UniversalCamerawork が定義するカメラ状態の共通フォーマット |
| Global Volume | URP における、シーン全体に適用される Volume（ポストプロセス等） |
| Local Volume（カメラ Volume） | 特定カメラ／特定エリアに適用される URP Volume |

## 10. オープンイシュー（設計フェーズで確定すべき項目）

本要件定義の段階では固めていないが、設計フェーズまでに決定が必要な項目を列挙する。

- LocalHost 通信の具体トランスポート（TCP / UDP / WebSocket / NamedPipe）と
  スキーマ（JSON / MessagePack / Protobuf）。
- OSC によるカメラデータ送信のアドレスパターン・メッセージ構造の詳細。
- Light 構成、カメラ構成、グローバル Volume 設定の永続化形式とファイル配置。
- 起動時の一括ロード対象と、AssetBundle で遅延可とする対象の境界。
- ビルド時・Editor 時の通信チャネルの切り替え方法。
- UI スキンの差し替え粒度（USS のみ / UXML も差し替え可 / 両方）。
- RuntimeDisplaySelector の API 確定後の組み込み方針。
